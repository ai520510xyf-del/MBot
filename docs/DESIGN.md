## 1. 产品定位

### 1.1 一句话定位

> **口袋里的小龙虾 —— 下载 App 就拥有完整 AI Agent 系统，无需任何服务器部署**

### 1.2 核心价值

- **零门槛**：下载 App → 绑定微信/QQ → 开始聊天
- **完全本地**：Gateway 运行在手机上，数据不上传
- **隐私优先**：端侧模型可选，对话完全私密
- **能力扩展**：5000+ 技能生态，AI Agent 级别智能

### 1.3 核心理念转变

```
❌ 旧设计：App → 连接云端 Gateway → 调用模型
✅ 新设计：App 内嵌 Gateway → 本地运行 → 连接外部通讯平台
```

## 2. 目标用户画像

### 2.1 核心用户

| 特征 | 描述 |
|------|------|
| **设备** | 只用手机，没有电脑或不想开电脑 |
| **年龄** | 18-45岁，广泛人群 |
| **技术能力** | 普通用户，不会命令行，不懂技术 |
| **通讯习惯** | 每天用微信/QQ/Telegram |
| **AI 认知** | 听说过 AI 想用，但不知道怎么入门 |
| **支付意愿** | 不愿买新设备，愿意为好用的小工具付费 |
| **隐私敏感** | 关注数据隐私，希望对话不被收集 |

### 2.2 用户痛点

```
痛点 1：想用 AI 但不知道怎么开始
       ↓
       ChatGPT 需要科学上网、注册复杂、不支持微信

痛点 2：已经在用的 AI 助手不够智能
       ↓
       只能聊天，不能执行任务，没有记忆

痛点 3：没有电脑，很多 AI 功能用不了
       ↓
       桌面端 AI 工具丰富，但手机端被阉割

痛点 4：担心 AI 收集隐私数据
       ↓
       能不能有一个本地运行、不上传对话的 AI？

痛点 5：每次都要打开 App
       ↓
       能不能直接在微信里和 AI 对话？
```

### 2.3 使用场景

| 场景 | 用户行为 | QClaw 价值 |
|------|---------|-----------|
| 日常咨询 | 微信里问"今天天气" | 不用切换 App |
| 学习辅导 | QQ 里问"解释这段代码" | AI Agent 深度分析 |
| 工作提效 | Telegram 里让 AI 帮写文案 | 技能扩展能力 |
| 生活助手 | 微信里"帮我查快递" | 记忆 + 工具调用 |
| 碎片时间 | 任何地方随时和 AI 聊天 | 多通道统一记忆 |
| 隐私对话 | 本地模型处理敏感内容 | 完全私密，不上传 |

### 5.2 关键架构决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| **Gateway 位置** | 集成在 App 内，手机本地运行 | 用户无需部署，零门槛，完全隐私 |
| **Node.js 集成** | nodejs-mobile 嵌入式运行时 | OpenClaw Gateway 原生支持 |
| **模型选择** | 云端 API + 端侧模型可选 | 性能与隐私平衡 |
| **通道实现** | Bot API 连接外部平台 | 稳定、合规、可大规模推广 |
| **状态管理** | 本地 SQLite + 本地向量库 | 完全本地，数据不上传 |
| **数据存储** | 本地 SQLite | 隐私优先，可选云端同步 |

### 5.3 技术挑战与解决方案

| 挑战 | 解决方案 |
|------|----------|
| **手机运行 Node.js** | nodejs-mobile 嵌入式运行时 |
| **Flutter ↔ Node.js 通信** | Platform Channel 双向通信 |
| **端侧模型推理** | llama.cpp / MLX (iOS) / MediaPipe |
| **资源消耗控制** | 按需启动 Gateway，智能休眠 |
| **多通道并发** | 事件驱动架构 + 消息队列 |

## 6. 技术选型

### 6.1 移动端框架

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **UI 框架** | Flutter 3.x | 跨平台、高性能、热重载 |
| **语言** | Dart 3.x | Flutter 官方语言 |
| **状态管理** | Riverpod 2.x | 类型安全、测试友好 |
| **本地存储** | Drift (SQLite) | 强类型、迁移友好 |
| **网络** | dio + web_socket_channel | HTTP + WebSocket 支持 |
| **UI 组件** | Material 3 + 自定义主题 | 现代化设计 |
| **推送** | Firebase Cloud Messaging | 跨平台推送 |
| **语音** | flutter_tts + speech_to_text | 基础语音能力 |

### 6.2 嵌入式运行时

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **Node.js 集成** | nodejs-mobile | 官方支持，稳定 |
| **Gateway 框架** | OpenClaw Gateway | 与 QClaw 桌面端统一 |
| **JS 运行时** | Node.js 20+ / Bun | OpenClaw 原生支持 |
| **通信桥接** | Platform Channel | Flutter ↔ Native 通信 |
| **内存管理** | 智能回收 + 按需加载 | 控制资源消耗 |

### 6.3 端侧模型（可选）

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **iOS 推理** | MLX / Core ML | Apple 原生支持 |
| **Android 推理** | llama.cpp | 跨平台，高性能 |
| **模型格式** | GGUF | 压缩率高，加载快 |
| **推荐模型** | Qwen2.5-1.5B / Phi-3-mini | 小而美，端侧友好 |

### 6.4 云端模型接入

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **LLM 接入** | OpenAI 兼容 API | 支持多家模型提供商 |
| **推荐模型** | DeepSeek / 通义千问 / GLM | 国内可用 |
| **向量存储** | 本地向量库 (SQLite 扩展) | 完全本地 |
| **缓存** | 内存缓存 + 本地持久化 | 快速响应 |

### 6.5 通道连接

| 通道 | 技术方案 | 状态 |
|------|---------|------|
| **微信** | 第三方 Bot API | 需申请 |
| **QQ** | NapCat / Lagrange / 官方 Bot API | 开源优先 |
| **Telegram** | Telegram Bot API | 免费、易用 |
| **飞书** | 飞书开放平台 Bot API | 企业场景 |

## 8. 风险评估与缓解

### 8.1 技术风险

| 风险 | 等级 | 影响 | 缓解方案 |
|------|------|------|---------|
| **nodejs-mobile 稳定性** | 高 | 核心功能不稳定 | 1. 充分测试<br>2. Fallback 方案<br>3. 降级策略 |
| **手机资源消耗** | 高 | 卡顿、发热、耗电 | 1. 按需启动 Gateway<br>2. 智能休眠<br>3. 资源监控 |
| **端侧模型性能** | 中 | 响应慢、体验差 | 1. 云端模型优先<br>2. 模型量化<br>3. 异步推理 |
| **微信接入限制** | 高 | 核心功能无法使用 | 1. 多通道并行<br>2. 网页版 Fallback<br>3. 持续跟进 |
| **数据迁移** | 中 | 用户数据丢失 | 1. 导出/导入功能<br>2. 云端备份选项 |
| **跨平台兼容** | 中 | iOS/Android 差异 | 1. 统一接口<br>2. 平台适配层 |

## 10. 待决策问题

### 10.1 商业模式

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **免费 + Pro 订阅** | 基础免费，高级功能付费 | ✅ 用户门槛低<br>⚠️ 转化需时间 |
| **按量付费** | 按对话次数收费 | ✅ 公平<br>⚠️ 用户可能不敢用 |
| **Freemium** | 免费额度 + 用量付费 | ✅ 灵活<br>⚠️ 计费复杂 |

**推荐**：免费基础功能 + Pro 订阅（¥9.9/月）
- Pro 功能：无限制对话、高级技能、更大记忆、端侧模型
